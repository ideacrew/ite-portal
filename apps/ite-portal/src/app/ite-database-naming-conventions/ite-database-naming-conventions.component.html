ITE Naming Conventions

  • Use full descriptive words whenever possible (e.g., provider_type_code, diagnosis_code, etc.) Use an abbreviation, only when it's commonly used or the word is too long. In those cases, any abbreviations used should be added to the data dictionary (e.g., smi, sed, cafas, etc.)
  • Do NOT use 'space' or any special characters in any object names.
  • Use the snake_case rule: write all characters in lowercase with an underscore for composite names, to name all database objects, including databases, schemas, tables, views, and column names. This is per naming convention rules governed by Ruby on Rails, which is one of the most popular web application development frameworks, used to develop the ITE portal and Provider Gateway.
  • Table names should be plural (e.g., clients, episodes, diagnoses, etc.)
  • Column names should be singular_snake_case (e.g., first_name, date_of-birth, claim_id, etc.)
  • A foreign key column should be singularized_table_name (e.g., client_id for clients table, claim_id for claims table, address_id for addresses table, etc.)
  • Data analysts or scientists creating data reports or dashboards may translate column names into different words or in different styles that may be more accustomed to the business or the audience of the data reports. In that case, the original names of the source objects will not change, and it is recommended to document translations whenever possible.
  • Apply the same rules consistently.
